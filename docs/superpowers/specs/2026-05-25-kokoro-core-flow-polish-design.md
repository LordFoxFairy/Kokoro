## 1. 背景

Kokoro 当前已经完成一轮高保真静态原型，产品人格、应用功能与案例库的分离、Studio 空态与结果页双态、以及“我的任务 / 调度能力”都已初步建立。但当前原型仍然存在明显的不均衡：

1. **页面单看成立，系统感还不够强**：页面之间已有链接，但用户未必能自然理解完整闭环。
2. **完成度不均衡**：有些页面已接近成熟展示页，有些仍偏“原型说明感”或“功能接通感”。
3. **视觉语言已成立，但精修度不足**：层级、留白、动作轻重、重点与次重点的区分尚未打磨到成品级。
4. **核心回路已经存在，但还不够流畅**：从表达需求到产出结果，再到持续跟进与案例复用，这条链路还没有完全“活起来”。

因此，下一轮不宜把“全站所有页面”作为一个一次性任务粗暴推进，而应先围绕 **核心主链路** 做系统级全面打磨。

---

## 2. 设计结论

**结论一句话：**

> Kokoro 下一轮打磨的重点，不是“再补更多页面”，而是把核心主链路从“可演示原型”打磨成“系统感明确、节奏连续、成熟可信的产品原型”。

这个主链路由 5 个核心触点组成：

1. 首页 / Chat
2. 应用功能页（Studio）
3. 结果页（Result）
4. 我的任务
5. 案例库

这 5 个触点共同构成 Kokoro 最重要的产品循环：

> 表达需求 → 开始创作 → 得到结果 → 继续协作 → 沉淀方法 → 再次开始

---

## 3. 本轮打磨目标

### 3.1 核心目标

1. **强化主链路连续性**：让页面之间不只是“可跳转”，而是“逻辑上自然接续”。
2. **提升成品感**：让关键页面从“原型说明页”进化成“可信产品页”。
3. **统一信息层级**：每个页面都能一眼看出主动作、当前状态和下一步。
4. **让调度能力真正融入产品，不像外挂功能**。
5. **提高整体现代感与温度感的平衡**：保持 Kokoro 的米木纸感与陪伴气质，同时降低粗糙原型感。

### 3.2 非目标

本轮不做：

1. 全站所有边缘页一次性重磨
2. components / interactions / gallery 等设计系统展示页的优先重构
3. admin / agent / workflow 等次级或高级页的深度设计
4. 真实实现逻辑、接口、状态机、任务引擎等工程层工作
5. 复杂的视觉重品牌或产品定位重置

---

## 4. 为什么只先打磨核心主链路

### 4.1 因为这是产品真正被感知的地方

用户对 Kokoro 的第一印象，不来自“管理后台页”或“边缘设置页”，而来自：
- 我怎么开始说话
- 它怎么理解我
- 我怎么开始做图 / 做文档 / 做站点
- 它怎么给我结果
- 结果之后它是否还能继续跟我协作

### 4.2 因为这条链路最能体现 Kokoro 的产品差异

Kokoro 不是简单工具集，也不是纯聊天机器人。它最重要的差异，在于：
- 有温度的人格
- 面向创作者与新手创业者
- 结果可视化
- 可持续协作

这些都集中体现在主链路上。

### 4.3 因为先做这 5 个触点，后续全站扩展会自然很多

一旦这 5 个点做成熟，后续去处理：
- settings
- library
- teams
- search
- skill hub / mcp hub
- agents / workflows

就会有明确标准可套，而不是每页单独重想一次。

## 5. 本轮打磨对象与职责分工

### 5.1 首页 / Chat

#### 角色
承担“表达需求、感受 Kokoro 气质、进入协作”的入口角色。

#### 当前问题
- 有气质，但局部仍偏“演示式排布”；
- 对话中的系统行为、工具行为、结果跳转关系还不够精细；
- 调度确认、继续动作、回到任务中心这些新能力虽然已出现，但尚未完全和对话语气融成一体。

#### 打磨重点
- 对话的阅读节奏
- 工具调用信息的轻重层级
- 调度确认卡的融合感
- 从 chat 去 result / task center 的自然性
- 顶部与底部输入区的“真实产品感”

---

### 5.2 应用功能页（Studio）

#### 角色
承担“开始创作”的统一入口角色。

#### 当前问题
- 单列生成器方向是对的，但不同 Studio 间还不够像同一套成熟系统；
- 调度入口已加上，但仍需要从“附加控件”变成“自然的时间扩展动作”；
- 某些页面的主体留白、主信息、辅助说明还不够均衡。

#### 打磨重点
- Hero 区块的识别度与统一度
- 生成栏的产品感与交互密度
- “生成”与“稍后做”的主次关系
- 参数 chips 的秩序感
- 空态与即将生成之间的情绪连续性

---

### 5.3 结果页（Result）

#### 角色
承担“结果交付 + 继续下一步”的角色。

#### 当前问题
- 一些结果页更像单次展示，不像持续协作入口；
- 新增的“明天继续 / 每周继续”方向是对的，但需要更深入变成页面结构的一部分；
- 结果页中的聊天区、画布区、操作区层级还可以继续压实。

#### 打磨重点
- 结果页主结果的舞台感
- 次级动作的排布
- “再做一版 / 明天继续 / 每周继续”的回路感
- 结果与任务中心的关系
- 结果页是否能承担“沉淀为方法”的入口

---

### 5.4 我的任务

#### 角色
承担“持续协作总览”的中枢角色。

#### 当前问题
- 第一版结构已建立，但仍偏“新增页”而不是“系统中心”；
- 页面虽然已能看，但信息密度与任务卡完成度还可以继续打磨；
- 详情面板、分区、摘要卡之间仍可更精致地拉开层级。

#### 打磨重点
- 页面作为第二主场的稳重感
- 四大分区的辨识度
- 卡片的成果感与非后台感
- 任务详情面板的可信度
- 与通知、结果页、chat 的双向关系表达

---

### 5.5 案例库

#### 角色
承担“方法沉淀与二次启动”的角色。

#### 当前问题
- 已从模板库走向案例库，但还可以更明确地表达“方法墙”而不是“作品墙”；
- 新增的“明天来一版 / 每周按这个做”是正确方向，但还可继续提升卡片层级与引导力；
- 与应用功能页、结果页、任务中心之间的联系仍可更显性。

#### 打磨重点
- hero 的产品叙事清晰度
- 卡片的缩略、标题、动作之间的层级关系
- “现在试试”与“交给 Kokoro”之间的平衡
- 案例作为方法模板的可信度

---

## 6. 系统级问题：本轮必须统一的 6 件事

### 6.1 主动作永远清楚
每个页面必须一眼看出“最应该做的事”。

- chat：继续说 / 接住建议
- studio：生成
- result：看结果并决定下一步
- 我的任务：查看正在发生什么，决定是否接手
- 案例库：现在试，或交给 Kokoro 继续做

不能让页面出现多个同等级大动作，导致注意力分散。

### 6.2 每页都要能说明“下一步是什么”
原型常见的问题是：页面能展示很多东西，但没有明确的下一步。

本轮要求每个核心页面都能回答：
- 用户接下来最自然要去哪？
- 这个出口是否一眼可见？

### 6.3 同一能力在不同页面里的表达要一致
例如调度能力：
- 在 chat 里是确认卡
- 在 studio 里是“稍后做”
- 在 result 里是“明天继续 / 每周继续”
- 在 case library 里是“明天来一版 / 每周按这个做”
- 在 task center 里是持续总览

这些表达可以不同，但不能互相像不同产品。

### 6.4 页面不是说明图，而是生活中的产品界面
判断标准是：
- 是否像用户真的会长期使用的界面
- 是否有真实的信息轻重关系
- 是否看起来像“会活下去的产品”，而不是展板

### 6.5 页面之间要形成“节奏”而不是“跳转”
也就是：
- 从 chat 到 studio，不只是点链接
- 从 studio 到 result，不只是生成成功
- 从 result 到 task center，不只是多一个入口
- 从 case library 回到 studio / task center，也不只是导航

它们应该像一条连续的呼吸链。

### 6.6 所有打磨都必须服务 Kokoro 的“心”路线
继续坚持：
- 米、木、纸感
- 温和、克制、可信
- 不冷酷、不赛博、不毛玻璃、不炫技

但同时注意：
- 温柔不等于没有层级
- 克制不等于没有重点
- 暖色不等于松散

## 7. 页面级打磨原则

### 7.1 首页 / Chat 的打磨原则

1. 对话可读性优先于花哨装饰
2. 每一条 AI 输出都要看得出“它在做什么”
3. 工具调用信息要轻，不抢正文，但也不能太弱
4. 调度卡、结果卡要像聊天中的自然延续，不像系统插片
5. 输入区要更像长期使用的主控件，而不是演示型底栏

### 7.2 Studio 页的打磨原则

1. Hero 保持品牌识别
2. 生成栏必须成为真正的视觉中心
3. 调度入口只能是时间扩展，不可喧宾夺主
4. 参数项的组织要更专业、更静定
5. 整个页面既要像创作工具，也要像会持续协作的助手入口

### 7.3 Result 页的打磨原则

1. 主结果必须有舞台感
2. 次级动作要围绕“继续这条线”组织
3. 结果不是终点，而是新任务和新方法的起点
4. 页面要兼具“看结果”和“继续协作”两种语义

### 7.4 我的任务页的打磨原则

1. 像“被托付的事”，不是“系统任务”
2. 强调节奏、结果、状态与下一步，而不是技术字段
3. 让页面更像“委托墙 / 总览页”，而不是后台面板
4. 用结构感建立信任，不用密度堆出复杂度

### 7.5 案例库的打磨原则

1. 更像方法库，不只是展示库
2. 卡片既要能看，也要能启动行动
3. “现在试试”和“交给 Kokoro”要清楚区分层级
4. 每张卡都要有“这是可复用做法”的感觉

## 8. 本轮要解决的设计症状

这部分不是功能点，而是具体“产品看起来还不够成熟”的症状。

### 症状 1：信息都在，但重点不够突出
解决方向：
- 拉开标题 / 正文 / 辅助信息的层级
- 压缩不必要的视觉竞争
- 让用户主视线更稳定

### 症状 2：页面像拼起来的，不像呼吸一致的系统
解决方向：
- 统一动作命名
- 统一卡片节奏
- 统一状态表达
- 强化页面之间的关系链

### 症状 3：有些块像功能说明，不像真实 UI
解决方向：
- 减少“说明口吻”
- 提高“真实产品语气”
- 更注意控件位置、密度、留白与真实操作顺序

### 症状 4：新增能力接进来了，但还像外挂
解决方向：
- 调度能力要嵌入主链路
- 任务中心要成为第二主场
- 结果页和案例库要自然承接节奏能力

## 9. 打磨后的理想体验

如果这轮打磨成功，理想状态应当是：

### 用户第一次进入时
会觉得：
- Kokoro 很温柔，但不是软弱
- 它像一个会做事的产品，不只是聊天

### 用户开始创作时
会觉得：
- 页面很干净
- 主动作很明确
- 想立即做，或者想稍后做，都很自然

### 用户拿到结果时
会觉得：
- 结果被认真交付了
- 接下来还能继续走，不会断在这里

### 用户回来看任务时
会觉得：
- Kokoro 真的在替我记着事
- 这不是系统后台，而是我的持续协作面板

### 用户看案例库时
会觉得：
- 这些不是“别人做的作品”，而是“我也可以用的做法”
- 而且我可以马上试，或者以后继续做

## 10. 本轮设计输出要求

本轮设计不只给出“方向”，还应指导后续原型精修。输出必须覆盖：

1. 每个核心页面的角色重新定义
2. 每个页面的主要问题与重点打磨目标
3. 系统级一致性要求
4. 页面之间的连续关系
5. 文案与动作命名原则
6. 波次实施顺序

---

## 11. 原型精修顺序建议

推荐顺序：

### 第一优先级
1. Chat
2. 一个代表性 Studio 页
3. 一个代表性 Result 页
4. 我的任务
5. 案例库

### 第二优先级
将第一优先级形成的设计语言，扩散到：
- 其他 Studio / Result 页
- 首页
- 局部状态页

### 第三优先级
再处理系统支撑页：
- settings
- library
- teams
- search
- hub 页

原因：先把产品的“心脏”打磨好，再推广到全身。

---

## 12. 文案与动作命名统一原则

### 12.1 调度相关
统一优先使用：
- 稍后做
- 明天继续
- 每周按这个做
- 去我的任务
- 做完告诉我

避免混用：
- 定时任务
- 自动化
- 排程
- cron
- job

### 12.2 任务相关
统一优先使用：
- 我的任务
- 即将开始
- 正在进行
- 等你决定
- 已完成
- 暂停中
- 没做成

### 12.3 结果相关
统一优先使用：
- 再做一版
- 继续这条线
- 按这个风格来一版
- 看结果

---

## 13. 本轮之后的自然下一步

当这份设计稿被确认后，下一步不是立刻“全站一起改”，而是：

1. 写一份对应的实施计划（只针对核心主链路精修）
2. 先选一页 Chat、一页 Studio、一页 Result、我的任务、案例库作为代表页
3. 用这些页先做出新的设计基准
4. 再把基准推广到其它同类页面

这能确保打磨是“深度优先”，而不是“覆盖优先”。

## 14. 一句话收束

> Kokoro 下一轮最重要的工作，不是增加更多页面，而是把“说出需求 → 开始创作 → 拿到结果 → 持续协作 → 沉淀方法”这条核心主链路，打磨成一个真正连贯、成熟、可信的产品原型系统。
