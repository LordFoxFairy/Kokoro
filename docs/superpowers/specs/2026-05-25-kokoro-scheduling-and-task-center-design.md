# Kokoro 调度能力与任务中心设计稿

- **日期**：2026-05-25
- **状态**：draft

---

## 1. 背景

Kokoro 当前已经完成一轮高保真静态原型，产品人格、目标用户、主信息架构、应用功能页与结果页两态关系已经基本清晰。但当前原型仍然更像“当下生成一次的创作工具”，缺少“在未来时间点或固定节奏中持续替用户推进事情”的能力表达。

这会带来三个明显缺口：

1. **Agent 感不足**：如果 Kokoro 只能立即响应，它更像高级生成器，而不是持续协作型 agent。
2. **任务生命周期不完整**：从创建、等待、执行、确认、完成、回看，到下一次继续的链路没有被系统化表达。
3. **创作者节奏价值没有被接住**：内容创作者与新手创业者需要的不只是“现在做”，还需要“明早做”“每周做”“做完告诉我”。

因此，本设计稿定义 Kokoro 的“调度能力”与“任务中心”，作为下一轮产品原型打磨的核心能力层。

---

## 2. 设计结论

**结论一句话：**

> Kokoro 不应该把 cron 做成技术功能，而应该把它做成“把事情交给 Kokoro，它会替你按时间和节奏持续推进并把结果带回来”的产品能力。

这意味着：

- 前台不强调 cron / expression / scheduler config 等技术概念；
- 底层可以按标准调度系统能力来思考；
- 用户面对的是“稍后做、明早做、每周做、做完告诉我、按这个节奏继续”；
- 页面设计重点是“持续委托感”和“结果回流”，不是任务表格化管理。

---

## 3. 设计目标

### 3.1 目标

1. **让 Kokoro 从即时工具升级为持续协作对象**。
2. **让用户可以自然创建一次性或周期性任务**，而不用学习技术化的调度配置。
3. **补齐任务生命周期**：创建 → 等待 → 执行 → 等待确认 → 完成/没做成 → 回看 → 再次继续。
4. **把结果页、案例库、应用功能页、chat 串成闭环**。
5. **让这块能力符合 Kokoro 的“心”路线**：温和、清楚、克制、有陪伴感。

### 3.2 非目标

本轮原型**不做**：

1. 可视化 workflow builder
2. 开发者式 cron 表达式配置器
3. 运维式 job table / execution log 后台
4. 复杂企业级协同调度系统
5. 细到实现层的 provider / queue / retry 策略设计

---

## 4. 用户心智模型

Kokoro 的调度能力不应让用户形成“我在创建一个 job”的认知，而应形成下面这句话：

> 我把一件事交给 Kokoro，它会在合适的时候提醒我、替我开始、继续推进，或者把结果带回来给我。

这个心智模型分三层：

1. **当下创作**：现在生成、现在整理、现在开始。
2. **未来继续**：稍后做、明早做、下周提醒我。
3. **长期代办**：每周帮我做、按这个节奏持续跟进。

其中第三层是 Kokoro 与普通创作工具拉开差距的关键。

---

## 5. 能力分层

### 5.1 层一：提醒型任务

定义：到点提醒用户，但不自动生成最终产物。

典型表达：
- 明晚提醒我发这条内容
- 明天早上提醒我继续这个课件
- 周五提醒我复盘

价值：
- 低门槛
- 符合 Kokoro 的陪伴感
- 为后续更强执行型任务建立信任

### 5.2 层二：执行型任务

定义：到点由 Kokoro 自动开始或继续，完成后带回结果。

典型表达：
- 明早帮我整理一版公众号稿
- 每周一给我 3 个内容选题
- 每周五帮我生成一版复盘卡片

价值：
- 真正体现 agent 感
- 把创作从单次生成升级为持续产出
- 与结果页、案例库、分享增长直接挂钩

### 5.3 层三：条件触发型任务

定义：由事件而不是时间触发的轻自动化。

典型表达：
- 上一步完成后继续下一步
- 新素材入库后自动出一版草稿
- 没做成时再试一次并告诉我

本轮策略：
- 只允许在设计描述中轻度暗示它的存在；
- 不在本轮原型中做重；
- 不设计节点编排器，不做开发者工作流体验。

## 6. 页面与入口分布

调度能力不能做成一个孤立的“定时任务页”，而应长在现有主流程中。推荐接入 5 个触点。

### 6.1 Chat：自然语言创建任务

Chat 是最自然的入口。用户会直接说：
- 明早帮我整理成公众号版本
- 每周一给我三个选题
- 晚点提醒我继续这个海报

Chat 的职责：
- 识别时间 / 节奏 / 未来执行意图；
- 区分“提醒我”还是“替我做”；
- 通过轻确认卡创建任务。

### 6.2 应用功能页：生成动作的延伸

Image / Video / Design / Courseware / Landing 等应用功能页中，调度是“生成”动作的时间扩展，而不是独立流程。

应用功能页的职责：
- 保持“生成”是主动作；
- 在主按钮附近提供小时钟 / 节奏入口；
- 让用户把“现在生成”改成“稍后生成 / 定期生成 / 到时候提醒我”。

### 6.3 案例库：从灵感库升级为方法库

案例库不应只承担“看案例”和“现在试试”，还应支持：
- 明天来一版
- 每周按这个做

案例库的职责：
- 把案例从展示内容升级成可持续托付的方法；
- 让案例进入节奏性创作链路。

### 6.4 结果页：从终点变成下一次的起点

结果页不只是“看成品”，还应支持：
- 再做一版
- 明天继续
- 每周按这个风格来一版
- 把这次结果设成固定任务

结果页的职责：
- 承接完成通知；
- 支持从单次结果进入持续关系；
- 形成任务中心 ←→ 结果页的回环。

### 6.5 任务中心：总览所有委托

任务中心是这块能力的中枢页面，用来承接所有“被交给 Kokoro 的事”。

---

## 7. 任务中心在信息架构中的位置

推荐在左侧主导航中加入：

- 新对话
- 搜索
- 应用功能
- 案例库
- **我的任务**
- 团队
- 库·收藏

命名建议固定为 **“我的任务”**，理由：
- 用户容易理解；
- 比“自动化”“调度中心”“cron”更贴近当前产品气质；
- 既能承接提醒型任务，也能承接执行型任务。

“我的任务”不是首页，而是 Kokoro 的“第二主场”：一旦用户开始持续使用，它会成为高频回访页。

## 8. 任务中心：页面级原型定义

### 8.1 页面角色

任务中心页 = Kokoro 替用户记着、推进、等待回应、已经完成的所有委托的总览页。

它不是：
- 后台控制台
- 系统日志页
- cron jobs 管理页

它应该是：
- 持续工作的总览页
- 用户委托的聚合页
- 结果与节奏关系的总控页

### 8.2 页面结构

推荐采用“上轻下重”的结构：

#### 顶部：轻摘要
- 页面标题：**我的任务**
- 副标题：**这些是 Kokoro 正在替你记着和推进的事。**
- 4 个轻摘要数字：即将开始 / 正在进行 / 等你决定 / 本周已完成

#### 中部：4 大分区卡片流
1. 即将开始
2. 正在进行
3. 等你决定
4. 已完成 / 暂停中 / 没做成

#### 详情层
- 第一版原型建议用右侧详情抽屉或大浮层；
- 不急着新做独立“任务详情页”；
- 用户点卡片后就地查看细节、结果和动作。

### 8.3 轻量过滤

第一版只保留以下过滤：

- 来源：全部 / 来自对话 / 来自应用功能 / 来自案例库
- 节奏：全部 / 一次 / 每天 / 每周
- 状态：全部 / 进行中 / 等我决定 / 已完成

本轮不做：模型、成本、provider、技术日志、执行深层状态等过滤。

---

## 9. 任务中心四大分区定义

### 9.1 即将开始

心智：我已经交代好了，它很快就会发生。

卡片重点：
- 任务名
- 下一次发生时间
- 是提醒还是执行
- 来源
- 节奏
- 快捷动作：查看 / 现在执行 / 改时间 / 暂停

示例：
- 每周内容选题
- 下次：周一 09:00
- 来自：案例库 · 选题模板
- 节奏：每周

### 9.2 正在进行

心智：事情正在推进，Kokoro 没忘。

卡片重点：
- 正在做什么
- 做到哪一步了
- 是否已有中间结果
- 快捷动作：查看进展 / 暂停

示例：
- 给妈妈讲 AI 的课件整理
- 正在补第 1 版封面与目录
- 来自：课件功能

要求：
- 不只显示“执行中”三个字；
- 每张卡都应有一条具体过程句子。

### 9.3 等你决定

心智：Kokoro 已经推进到一个节点，现在轮到用户。

卡片重点：
- 已准备好的内容是什么
- 现在需要用户做什么选择
- 继续后会发生什么

示例：
- 小红书海报草案
- 我先准备了 3 个方向，等你挑一个继续。
- 动作：去看看 / 稍后提醒我

这一区必须保留，因为它与 Plan / Default / Auto 信任模型紧密相关。

### 9.4 已完成 / 暂停中 / 没做成

这一组是“成果归档”，不是垃圾历史。

#### 已完成
重点：结果预览 + 下一步动作

#### 暂停中
重点：为什么暂停 + 何时恢复 + 一键恢复

#### 没做成
重点：说明没做成的节点 + 再试一次 / 改一下再试

文案原则：优先用“没做成”，不用生硬的“失败”。

---

## 10. 任务卡片设计规范

每张任务卡至少包含以下元素：

1. **人话任务名**：例如“每周内容选题”“小红书封面草案”
2. **来源**：来自对话 / 应用功能 / 案例库 / 上次结果页继续
3. **节奏**：一次 / 每天 / 每周 / 工作日 / 自定义
4. **核心状态信息**：下次执行时间 / 正在整理第 1 版 / 等你选方向 / 刚刚完成
5. **结果预览**：缩略图、封面、文案首句、课件封面等轻量预览
6. **轻量动作**：查看 / 改时间 / 暂停 / 恢复 / 再做一版 / 立即执行

限制：
- 一张卡最多保留 2–4 个主动作；
- 不在卡片主视图展示技术字段；
- 不展示 cron 表达式。

## 11. 任务详情抽屉定义

点开任务卡后，第一版原型用详情抽屉承载深一层信息。

### 11.1 任务概览
- 任务名
- 来源
- 创建时间
- 当前状态
- 节奏
- 下次执行时间

### 11.2 任务协议说明
用人话说明：
- 我会在什么时候开始；
- 是提醒你还是替你直接做；
- 如果需要你确认，我会停在哪一步。

### 11.3 最近结果
- 最近一次产物预览
- 最近一次完成时间
- 跳转到对应结果页

### 11.4 轻时间线
采用“事件时间线”，而非技术日志：
- 上周一：给出 3 个选题
- 上周三：你暂停了这项任务
- 今天：改成每周五执行

### 11.5 操作
- 立即执行
- 改时间
- 暂停 / 恢复
- 再做一版
- 删除这项委托

本轮不做：
- 原始执行日志
- provider / token / 模型成本
- cron 表达式
- 后台 trace

---

## 12. Chat 调度确认卡定义

Chat 中的调度交互不是命令行，而是意图确认。

### 12.1 触发条件
当用户输入里出现：
- 时间点（明早、今晚、下周）
- 频率（每天、每周）
- 未来执行意图（到时候帮我做、提醒我）

就触发调度确认卡。

### 12.2 卡片结构

#### 上半部分：复述理解
例如：
- 我理解你想让我在明早 9:00 帮你整理第一版。
- 我理解你想让我每周一早上给你 3 个内容选题。

#### 下半部分：轻动作
- 就这样
- 改一下时间
- 只提醒我就好

若是周期任务：
- 每周一这样做
- 改节奏
- 先只做一次

设计要求：
- 第一层不要弹出复杂表单；
- 时间选择器只在“改一下”后出现；
- 优先自然语言 + 轻确认，而不是结构化配置先行。

---

## 13. 应用功能页的时钟入口定义

调度能力应依附“生成”动作，不重做主布局。

### 13.1 位置
- 主按钮：生成
- 辅入口：小时钟 / 节奏图标，位于生成按钮旁边

### 13.2 第一层弹层内容

#### 一次性
- 现在做
- 今天晚上
- 明天早上
- 自定义时间

#### 周期性
- 每天
- 每周
- 工作日
- 自定义节奏

#### 行为方式
- 到时候提醒我
- 到时候直接帮我做

设计要求：
- 让它看起来像“时间与节奏选择器”，不是调度后台；
- 主心智仍然是创作，不是自动化设置。

---

## 14. 结果页的继续动作定义

结果页必须从“终点”升级为“下一次节奏的起点”。

推荐在结果页提供以下动作：
- 再做一版
- 明天继续
- 每周按这个风格来一版
- 把这个设成固定任务
- 做完继续告诉我

结果页应支持三种回路：

1. **结果 → 再次生成**：继续迭代
2. **结果 → 设为周期任务**：持续产出
3. **结果 → 沉淀为案例方法**：反哺案例库

---

## 15. 案例库的节奏化定义

案例库应从“内容墙”升级为“方法墙”。

每张案例卡推荐提供：
- 现在试试
- 明天来一版
- 每周按这个做

这会让案例库不只是展示成果，而是承载“被验证过的方法 + 节奏性复用”。

---

## 16. 通知与回访链路

通知不应只是“任务已完成”，而应强调完成了什么，并把用户带回结果。

推荐通知表达：
- 我已经帮你整理好这周的 3 个选题了，要不要现在看第一版？
- 你定在今晚继续的课件，我已经补出了封面和目录。

通知终点优先级：
1. 对应结果页
2. 对应任务详情中的结果区
3. 若仍需确认，则回到“等你决定”态

这使通知成为“下一步入口”，而不是“消息已读”。

---

## 17. 与 Mode 模型的关系

调度能力必须服从 `ADR-003` 的信任档位。

### Plan
- 可以创建任务；
- 到关键执行前先给计划或草稿；
- 用户批准后继续。

### Default
- 一般步骤可自动推进；
- 关键外部动作或高成本动作前要确认。

### Auto
- 在授权范围内自动执行；
- 主要通过状态与结果回执来建立信心。

这意味着：定时任务不是单纯的“时间调度”，而是“时间 + 信任授权”的联合机制。

## 18. 文案原则

文案必须保持温和、清楚、克制，不技术化，也不过度卖萌。

### 推荐表达

#### 创建类
- 稍后做
- 明早做
- 到时候提醒我
- 按这个节奏继续
- 让 Kokoro 持续跟进

#### 状态类
- 即将开始
- 正在进行
- 等你决定
- 已完成
- 暂停中
- 没做成

#### 结果类
- 做完告诉我
- 有结果时提醒我
- 我先帮你出第一版
- 你回来时我会接着做

### 避免表达
- cron
- expression
- scheduler config
- execution failed
- job queue
- automation pipeline

## 19. 本轮原型范围建议

### 19.1 必做
1. **任务中心页**
2. **Chat 调度确认卡**
3. **一个应用功能页的时钟入口样板**
4. **一个结果页的继续动作样板**
5. **案例库节奏化动作样板**
6. **完成通知 / 等待确认通知样式**

### 19.2 可后置
1. 多页面统一铺开时钟入口
2. 周期任务的连续历史视图加强
3. 结果转案例的更深交互

### 19.3 本轮明确不做
1. 技术化 cron 配置面板
2. 条件分支自动化编辑器
3. 开发者工作流图
4. 企业级协同调度后台

---

## 20. 原型打磨优先级

推荐顺序：

1. 任务中心页
2. Chat 调度确认卡
3. 一个应用功能页中的时钟入口
4. 一个结果页中的继续动作
5. 案例库卡片上的节奏动作
6. 通知样式统一

理由：
- 先立中枢；
- 再补入口；
- 再补结果回环；
- 最后统一表层语言。

---

## 21. 最终定义

Kokoro 的调度能力可定义为：

> 让用户把创作与整理任务交给 Kokoro，在未来时间点或固定节奏中持续推进，并在需要时请求确认、在完成时带回结果。

这是一个产品能力层，不是一个孤立功能页，也不是一个技术化 cron 系统外露界面。

---

本稿完成后，下一轮产品原型打磨应围绕以下内容展开：

1. 设计并细化“我的任务”页面原型
2. 补 chat 调度确认卡
3. 补应用功能页时钟入口
4. 补结果页继续动作
5. 补案例库节奏动作
6. 统一通知与状态文案

这样可以在不进入实现的前提下，把 Kokoro 从“当下创作工具”推进到“持续协作型创作 agent”的下一阶段。
