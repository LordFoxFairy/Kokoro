# 熔断器：永远会问一次的危险动作

> Permission mode 决定大多数事情打不打扰；熔断器决定**有一些事情无论你怎么设置 Kokoro 都会停下来再问你一次**。
> 直接借鉴 Claude Code 的 protected paths + circuit-breaker 设计（见 [claude-code/learnings/04](../../research/claude-code/learnings/04-safety-and-surfaces.md)）。CC 在 `bypassPermissions` 最宽松的模式下，`rm -rf /` 和 `rm -rf ~` 仍然 prompt——同理，Kokoro 也要给某些动作设这种"穿透所有 mode 的兜底"。
> 心路线下的二次确认不是吓人，是替你停一下。措辞参考 [voice-and-tone.md](../02-personality/voice-and-tone.md)。

---

## 一句话

**就算你切到 Auto 模式让 Kokoro 一路自动跑，下面这些动作发生时它仍然会停下来等你点头。** 这不是 mode 的一部分，是独立兜底。

---

## 1. Protected paths 清单（Kokoro 语境）

CC 的 protected paths 是 `.git` / `.vscode` / `.claude` 等"不该被自动改的本地路径"。Kokoro 不是命令行工具，所以 protected paths 在我们语境里是"不该被自动改的产品状态"。

| 类别 | 具体动作 | 为什么 |
|---|---|---|
| **账号** | 修改邮箱 / 密码 / 绑定手机 / 删除账号 | 一旦错就找不回 |
| **订阅** | 升级 / 降级 / 取消订阅 / 改付款方式 | 牵涉钱 |
| **已发布模板** | 删除已被 fork ≥1 次的公开模板 | 影响其他人的依赖 |
| **公开分享** | 撤回已经被分享的链接 / Canvas | 用户的别人那里可能已经在看 |
| **跨设备同步** | 让某一端的本地状态覆盖云端 / 反之 | 静默覆盖 = 数据丢失 |
| **导出 / 永久删除** | 一键清空全部对话 / Canvas / 媒体 | 不可撤回 |
| **第三方写权** | 让 MCP server 第一次执行"写"类操作（发邮件、发贴、删文件） | 副作用出 Kokoro 之外 |
| **大额操作** | 单次任务预计消耗 > 阈值的 token / API 配额 / 第三方调用次数 | 钱、配额 |
| **批量** | 一次删 / 改 ≥ N 个 Canvas / 历史项 | 操作幅度大于"误操作"该有的幅度 |
| **撤回不可逆动作** | 把已经发出去的邮件 / 帖子 unsend（如果接了这种 MCP） | 通常做不到，要明说 |

> 这清单要随产品变化迭代。新加任何一个"做错了会让用户难受"的功能，都先想一想要不要加进这张表。

## 2. Circuit breaker 触发后的 UI

### 通用形态

- 是个**modal**（不是 toast、不是行内确认）—— 强制用户视线聚焦
- 文案分三段：发生了什么 / 影响 / 两个明确选项
- 默认焦点在"取消" / "再想想"，不在"确认"——参考 macOS 系统级危险操作惯例
- 不允许用 Enter 直接确认（避免"啪啪点过去"）—— 关键动作要求点击具体按钮
- 不允许 5 秒内连续两次触发同一确认（防误操作 / 防恶意刷）

### 文案模板（心路线）

```
[发生了什么 · 一句话直白说]

[影响 · 一句话讲清后果]

[两个按钮：再想想 / 确认这件事]
```

### 三个实际例子

**例 1 · 删除账号**

```
你要删掉这个账号。

我会保留你的数据 7 天，万一改主意了能回来。7 天后我会真的删干净，包括备份。这之后我也找不回来了。

[ 再想想 ]    [ 确认删除 ]
```

**例 2 · 撤回已公开的模板**

```
这个模板有 12 个人 fork 过。

撤回后链接立刻失效，新的人看不到了。已经 fork 过的人那边的副本还是他们的，不会跟着消失——这件事我管不到。

[ 再想想 ]    [ 撤回这个模板 ]
```

**例 3 · 让本地状态覆盖云端**

```
你这台设备上有 3 个 Canvas 比云端旧。

按你的意思继续，云端这 3 个会被你本地这版覆盖。如果云端那一版是你在别的设备改的，那些改动会丢。

[ 再想想 ]    [ 用本地覆盖云端 ]
```

### Auto mode 下的特殊处理

- Kokoro 不在 Auto mode 里自己点过这个 modal —— 哪怕模型说"用户上次说过同意"
- 触发后 Kokoro 暂停当前任务，等用户回来；其他平行任务可以继续
- 输入框/侧栏显示"有一件事等你看一下"的轻提示，不打全屏推送
- 用户回来后看到任务在哪里停了 + 一段简短的上下文（不是从头再读一遍）

### 不做的事

- 不做 "下次不再提示" 复选框 —— 这正是熔断器要兜的事，不能让用户把它关掉
- 不做"30 秒倒计时自动确认" —— 危险操作不能用时间逼用户
- 不在 modal 文案里堆术语 —— 一句人话就够
- 不在 modal 里塞二级表单（"请告诉我们删除原因") —— 干净给走

## 3. 与 permission mode 的关系

**独立兜底。** 即：

| 维度 | Permission mode | Circuit breaker |
|---|---|---|
| 控制什么 | 一般动作问不问 | 一组特定危险动作的二次确认 |
| 用户能改吗 | 可改（4 档切换） | 不能关，只能改"哪些动作算危险"（设置里有限暴露） |
| 何时生效 | 每次工具调用前 | 在 mode 决定"不问"之后，仍然问 |
| 触发时 mode 会变吗 | 否 | 如果是 Auto 触发的，回退到 Ask 直到用户处理 |
| 适合的心智 | "我对 AI 的总体信任度" | "这件事错了就麻烦了" |

### 配合的方式

- mode = "通常我打扰你多频繁"（用户级）
- circuit breaker = "无论如何这几件事我都会停下来问"（产品级）

两层意图正交。一个是"调风险偏好"，一个是"防致命错误"。

## 4. 谁能改 circuit breaker 清单

### 普通用户

- 看到清单（设置 → 安全 → 这些事我总会问你一次）
- 不能删条目
- 可以**加**条目（自己觉得某动作敏感，要求 Kokoro 也熔断它）—— 自加只对自己生效

### 企业管理员

- 可以加全员条目（如 "调外部 webhook 必熔断"）
- 不能删 Kokoro 内置的条目（产品级红线）

### Kokoro 团队

- 通过 server-managed settings 推条目（参考 CC 的 managed settings）
- 加条目前公开变更记录

## 5. 熔断器误触发时怎么办

如果用户反复触发某条但每次都点确认（说明这件事对他不是"危险"，是日常）：

- 不"学习"后默认放行 —— 这正是熔断器要保的
- 但可以**提供针对该条的 mode**：例如"每天发 100 封邮件的销售用户"可以申请把"发邮件"从熔断器降为普通 ask
- 申请走人工 / 企业 admin —— 不开自动批的口子

### 反向：触发后用户不知所措

- modal 提供"我先不决定" —— 任务挂起，不强迫现场选
- 挂起的任务在 sidebar / 通知中心列出，随时回来处理

## 6. 测试规则（给 QA / dev 用）

每个 circuit breaker 必须验证：

- [ ] Ask / Accept edits / Plan / Auto 4 档下都触发
- [ ] Auto 下触发会暂停而不是失败
- [ ] modal 默认焦点不是"确认"
- [ ] 5 秒内重复触发被防抖
- [ ] modal 文案过了 [voice-and-tone.md](../02-personality/voice-and-tone.md) 的"三秒判断法"
- [ ] 移动端 modal 不被键盘遮挡 / 不能被滑掉
