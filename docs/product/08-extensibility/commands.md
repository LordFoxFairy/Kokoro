# Commands

> Kokoro **不**做独立的命令系统。「命令」= **手动触发的 skill**。
> 这是直接抄 Claude Code 的功课——他们把命令并入 skills 之后，整个扩展体系一下变干净了。

---

## 0. 立场

很多产品的「命令」和「预设」是两套：

- 命令：用户敲 `/foo`，背后是一段固定逻辑
- 预设：用户从模板库挑一个，套到对话里

Kokoro 拒绝走这条路。从用户视角看，**「我希望让 AI 做某件事的快捷方式」就是同一件事**，不该被拆成两套语法、两个管理界面、两份文档。

所以：

- 「命令」在 Kokoro 不是一个独立子系统
- 「命令」是 [skills](./skills.md) 的**手动触发面**
- 写一个新命令 = 写一个 skill 并填 `trigger.manual`

本文只描述「手动触发面」这层 UI/UX，skill 本体的 anatomy 看 [skills.md](./skills.md)。

---

## 1. 触发前缀：用什么

> ⚠️ **待你拍板**：用 `/` 还是别的字符。

候选：

| 前缀 | 优势 | 劣势 |
|---|---|---|
| `/` | 最通用（Slack / Discord / Notion / Claude Code / VSCode 全用），用户零学习 | 跟 URL 路径冲突——如果用户在写文档时输 `/path/to/file` 会误触发 |
| `@` | 不会跟路径冲突 | 跟 @-mention 概念在团队版会撞 |
| `:` | 简短 | 不直观 |
| `>` | 复古 / 终端感 | 视觉上不像"动作" |

**我的建议**：用 `/`，按以下规则消歧义：

1. 只在输入框**行首**触发命令补全（光标在中间打 `/` 不弹）
2. 命令字符必须是字母 / 数字 / 连字符（`/path/to/file` 因为下一字符是字母，且有第二个 `/`，可以模式化排除）
3. 任何命令补全都是非阻塞的——可以按 `Esc` 关掉补全继续写自由文本

这个判断不大不小，**等你拍板**。

## 2. 命令的形态

```
/journal                 # 无参数
/poster 主题=春日活动     # 命名参数
/translate en→zh         # 自由参数串
```

### 参数填法

skill 的 frontmatter 里声明参数：

```yaml
trigger:
  manual: /poster
  args:
    - name: theme
      label: 主题
      required: true
    - name: vibe
      label: 调性
      required: false
      options: [温柔, 严肃, 复古]
```

用户输 `/poster` 回车后：

- 有 required 参数 → 弹一个内联表单让填
- 没参数 → 直接进入 skill 主体

输 `/poster 主题=春日活动 调性=温柔` 也能直接带参数走。

### 不要做的

- ❌ 命令带复杂的 shell-like 语法（`--flag value`）。这是 CLI 的，不是聊天的
- ❌ 命令链（`/foo && /bar`）。要做组合就写一个 skill 把流程包进去
- ❌ 命令带 stdin/stdout 概念。Kokoro 不是终端

## 3. 自动补全 / 发现

用户在行首打 `/`，下方浮出补全面板：

```
/journal       整理一段碎碎念为结构化日记
/poster        做一张单页静态视觉物
/recap         本周复盘
/translate     语言翻译
/template      从模板新建 Canvas
─────────────
最近用过：
/journal       上次 2 小时前
─────────────
没找到？/help 看全部
```

排序逻辑：

1. 最近用过（个人维度）
2. 匹配度（输入字符 vs name 和 description）
3. 内置优先于自创
4. description 简短预览

### 发现入口

- 输入框行首 `/` → 即时补全
- `/help` → 全量列表 + 分组（内置 / 我的 / 团队 / 社区）
- 命令面板 `Cmd+K` → 全局搜索（命令、Canvas、对话、设置都在这）

## 5. 跟自由输入的区分

用户跟 Kokoro 互动有两种入口，必须让用户**感觉得到**它们的差异，但**不必死守**：

| | 自由输入 | 命令 |
|---|---|---|
| 长什么样 | 一段中文/英文自然话 | `/` 开头 |
| 给我的信息 | 你想干什么，让我自己想流程 | 你已经知道想用哪个流程，直接走 |
| 适合 | 想法还散，先聊聊 | 想法已收敛，直接做 |
| 触发 | 我可能自动激活某个 skill（v2） | 你显式调起 |

**重要**：自由输入和命令之间没有墙。

- 用户说「帮我做张海报」→ 我自动想到 `/poster` skill，可能直接走或先问一句
- 用户输 `/poster` → 进 skill 之后还是自然语言对话，不是表单填一遍就完事

这是 Kokoro 的态度：**命令是给「我已经想清楚要哪个流程」的快捷方式，不是给「让我跟 AI 做事更结构化」的工具**。后者交给自然语言。

## 6. 心人格细节

命令补全面板的微文案要服务心人格：

- 描述用第一人称（"我帮你..."），不要"系统将..."
- 找不到命令时 → "没找到这个。要不要直接说说你想做什么？"
- 命令执行失败 → "这个流程我跑不下去——\<原因\>。要不要换种方式？"
- 不要"命令未识别"、"无效命令"这种冷字眼

---

## 7. 反例：不要做成什么

- ❌ **不要做成 CLI**。Kokoro 是聊天，不是终端。命令是入口，不是主面
- ❌ **不要做成菜单驱动产品**。命令是给已知用户的快捷，不是新用户的引导
- ❌ **不要让用户「必须用命令」才能用某个功能**。命令必须有自然语言对应
- ❌ **不要给命令加 emoji 前缀**（违反心人格）
